模型配置方案对比 — 从引导到完全集成

三种实现深度：引导安装 cc switch → 唤起本地 cc switch → 完全内建

A

🔗 引导式 — 检测 + 引导安装 cc switch

App 检测 cc switch 是否可用 → 引导用户安装 → 用户安装后回到 App 自动识别。

优势

  • 实现最简单（1天内）
  • 零安全风险，凭据完全由 cc 管理
  • 配置文件格式兼容天然保证

代价

  • 依赖外部工具，用户体验割裂
  • 需切到终端操作，不是产品化体验
  • 未安装 cc switch 的用户卡住
B

🤝 桥接式 — 内嵌 cc switch 能力

App 自带 cc switch 核心逻辑，但保留其配置格式。用户在 App 内完成配置，同时与 CLI cc 互通。

优势

  • 原生 UI 体验，无需离开 App
  • 与 CLI cc 双向兼容（同配置文件）
  • 渐进式——先支持核心功能

代价

  • 需维护配置文件读写兼容层
  • cc switch 功能更新需跟进
  • 中等开发量（3-5天）
C

🚀 完全内建 — 模型设置彻底集成 ⭐ 推荐

App 完全取代 cc switch 的模型管理职责。自有配置文件 + UI 管理面板。同时兼容现有 cc 用户（导入兼容）。

优势

  • 最佳用户体验——全在 App 内完成
  • 不依赖任何外部工具
  • 可扩展 cc switch 不具备的功能
  • 最"产品化"的方案

代价

  • 开发量最大（5-7天）
  • 需设计自有配置格式 + cc 兼容导出
  • 密钥管理需极谨慎（Keychain）
