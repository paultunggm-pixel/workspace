千问小讲堂 · 挂载封面生成

工作流 / 产品需求 / 卡点与问题

面向：产品 设计 内容 工程
结论先行：这不是一个人能闭环的事。至少需要「内容 + 设计 + 工程 + 审校」四方各自定标准、各自交付， 否则像「成品图错名」这类错漏，无法被系统性地发现。

1我们在做什么

把每条课程内容（学名 / 钩子 / 插画）批量合成为统一品牌样式的竖版封面（144×192）， 并上传 OSS、把封面 URL 回写到内容表。已覆盖四类样式：

样式适用主文案字体
style1小低数学钩子（整段自动换行）造字工房圆黑 + 普惠
style2小高数学钩子（整段自动换行）普惠 Bold
style3语文（单名）学名HYXinRenWenSong 65W
成语故事成语 / 寓言学名（大）+ 钩子（小）65W 标题 + W 常规钩子（白 70%）

2标准工作流（现状）

① 内容表学名 / 钩子 / 干净插画 url / 年级
→
② Figma 规格字号 / 位置 / 字重 / 换行 / 配色
→
③ 渲染脚本选样式 → 排版换行 → 毛玻璃徽标 → 文字
④ 双版本产出PNG 无损 + JPG ≤50K 压缩
→
⑤ 上传 OSS得到 CDN URL
→
⑥ 回写内容表压缩前 url / 压缩后 url

其中部分篇目（如寓言故事）由设计老师直接出成品图，可跳过渲染直接进入 ④。 交付标准：每条两版 —— PNG 压缩前（无损，约 250–420KB）、JPG 压缩后（≤50KB，自适应画质）。

3产品需求 · 把样式当「规则」而非「个例」

  • 输入规范：内容表稳定提供 学名 / 钩子 / 干净插画 url / 年级或学科，列名统一、每条带唯一 guid。
  • 样式规则化：每种样式的字体/字号/字重/位置/行高/字间距/配色，由设计在 Figma 固化为「可读规格」，而非逐图量取。
  • 可读性规则：白字落在浅色插画上的处理（压暗 / 毛玻璃 / 阴影）需有明确判定标准。
  • 文案换行规则：钩子断行（按标点断、不显示逗号、避免孤字、两行均衡）在内容侧约定清楚。
  • 交付标准：压缩目标（30–50KB）、格式（JPG 有损 / PNG 无损）、命名与回写列名固定下来。

4卡点与问题 · 需各方认领

等级问题现状 / 例子影响责任方
高成品图错名 画龙点睛.png 实为塞翁失马、画龙点睛-1.png 实为惊弓之鸟；缺真正的画龙点睛 按文件名匹配会全错，需逐张开图人工核对设计出图 + 审校
高设计规格未沉淀 钩子字号/字重一度无人知，靠在 Figma 逐个点开文本框扒取 工程反复返工，规格易漂移设计
中可读性无标准 浅底白字发虚；压暗/毛玻璃/阴影选型来回试 出图质量不稳定设计
中文案换行无约定 钩子贪心换行出现「子？」孤尾；逗号是否显示不统一 排版观感差，需逐条调内容 + 设计
高数据列名不统一 学名确认版 vs 学名·确认版；部分表无封面 url 列 脚本需逐表适配，易错产品 + 内容
高匹配靠中文名 成品图 / 数据按「学名」匹配，命名不一致即丢失 漏配、错配（应改用 guid）全体
中覆盖不全 50 篇成语只有 20 张成品图，画龙点睛缺图 需「生成兜底」补齐设计 + 工程
中工程未产品化 OSS 凭证 / ossutil 路径本地硬编码；脚本单机跑 不可协作、不可复用、有安全隐患工程
高审校环节缺失 命名、文案、字体合规、可读性目前全靠临时人工抽检 错漏无法被系统发现新增审校 / QA
