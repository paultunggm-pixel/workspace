# 千问小讲堂 · 挂载封面生成 — 工作流与需求说明

> 面向：产品 / 设计 / 内容 / 工程
> 目的：说明现状工作流、产出标准、以及当前需要多角色协作的卡点与问题。
> 结论先行：**这不是一个人能闭环的事**，至少需要「内容 + 设计 + 工程 + 审校」四方各自定标准、各自交付，否则错漏（如成品图错名）无法被系统性发现。

---

## 1. 我们在做什么

把每条课程内容（学名/钩子/插画）批量合成为统一品牌样式的竖版封面（144×192 设计尺寸），并上传 OSS、把封面 URL 回写到内容表。

目前已覆盖的样式：

| 样式 | 适用 | 主文案 | 字体 |
|------|------|--------|------|
| style1 | 小低数学 | 钩子（整段自动换行） | 造字工房圆黑 + 普惠 |
| style2 | 小高数学 | 钩子（整段自动换行） | 普惠 Bold |
| style3 | 语文（单名）、科普/妙题 | 学名（有《》则仅内文） | HYXinRenWenSong 65W |
| 成语故事 | 成语/寓言 | 学名（大）+ 钩子（小） | 65W 标题 + W 常规钩子（白70%） |

## 2. 标准工作流（现状）

```
内容表(Excel/CSV)          设计稿(Figma)            设计成品(可选)
  学名/钩子/url/年级   →     规格(字号/位置/         →   部分篇目老师直接出图
                            字重/换行/配色)
        │                        │                        │
        └──────────┬─────────────┘                        │
                   ▼                                       │
           渲染脚本 render_covers.py  ◄────────────────────┘
        (选样式 → 排版换行 → 自适应毛玻璃徽标 → 文字)
                   ▼
        双版本产出：PNG(无损) + JPG(≤50K 压缩)
                   ▼
            上传 OSS(ossutil) → 得到 CDN URL
                   ▼
        回写内容表：压缩前url(PNG) / 压缩后url(JPG)
```

产出标准：
- 尺寸/样式按学科·年级自动匹配；品牌字体、徽标、文字排版统一。
- 每条交付两版：**PNG 压缩前（无损，约 250–420KB）**、**JPG 压缩后（≤50KB，自适应画质）**。
- 封面 URL 按内容回写到表，供线上挂载。

## 3. 产品需求（要把样式当「规则」而非「个例」）

1. **输入规范**：内容表需稳定提供 `学名` / `钩子` / `干净插画 url` / `年级或学科`，列名统一、每条带唯一 `guid`。
2. **样式规则化**：每种样式的字体/字号/字重/位置/行高/字间距/配色，需由设计在 Figma 固化为「可读规格」，而不是逐图量取。
3. **可读性规则**：白字落在浅色插画上的处理方式（压暗 / 毛玻璃 / 阴影）必须有明确判定标准，而非凭感觉。
4. **文案换行规则**：钩子的断行（按标点断、不显示逗号、避免孤字、两行均衡）应在内容侧约定清楚。
5. **交付标准**：压缩目标（30–50KB）、格式（JPG 有损 vs PNG 无损）、命名与回写列名固定下来。

## 4. 卡点与问题（重点 — 需各方认领）

| # | 问题 | 现状/例子 | 影响 | 建议责任方 |
|---|------|-----------|------|------------|
| 1 | **成品图错名** | `画龙点睛.png` 实为塞翁失马、`画龙点睛-1.png` 实为惊弓之鸟；且缺真正的画龙点睛 | 直接按文件名匹配会全错，需逐张开图人工核对 | 设计出图 + 审校 |
| 2 | **设计规格未沉淀** | 钩子字号/字重一度无人知，靠在 Figma 逐个点开文本框扒取 | 工程反复返工，规格易漂移 | 设计 |
| 3 | **可读性无标准** | 浅底白字发虚；压暗/毛玻璃/阴影选型来回试 | 出图质量不稳定 | 设计 |
| 4 | **文案换行无约定** | 钩子贪心换行出现「子？」孤尾；逗号是否显示不统一 | 排版观感差，需逐条调 | 内容 + 设计 |
| 5 | **数据列名不统一** | `学名确认版` vs `学名·确认版`；部分表无封面 url 列 | 脚本需逐表适配，易错 | 产品 + 内容 |
| 6 | **匹配靠中文名** | 成品图/数据按「学名」匹配，命名不一致即丢失 | 漏配、错配 | 全体（应改用 guid 关联） |
| 7 | **覆盖不全** | 50 篇成语只有 20 张成品图，画龙点睛缺图 | 需「生成兜底」补齐 | 设计 + 工程 |
| 8 | **工程未产品化** | OSS 凭证/ossutil 路径本地硬编码；脚本单机跑 | 不可协作、不可复用、有安全隐患 | 工程 |
| 9 | **审校环节缺失** | 命名、文案、字体合规、可读性目前全靠临时人工抽检 | 错漏无法被系统发现 | 新增审校/QA 角色 |

---

## 5. 建议的角色与分工

- **产品**：定输入规范、唯一 ID、交付标准、覆盖范围。
- **设计**：在 Figma 固化每样式规格 + 可读性判定规则；成品图按规范命名、补齐缺图。
- **内容**：保证学名/钩子/url 准确，约定钩子断行与标点规则。
- **工程**：维护渲染/压缩/上传/回写流程，改为按 guid 关联、列名规范化、凭证外置。
- **审校 / QA**：成品图开图核名、文案与字体合规、浅底可读性抽检、回写一致性校验。
